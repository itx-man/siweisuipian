Rust 所有权系统：从混乱到顿悟

学 Rust 三周，终于在某个深夜，突然理解了为什么借用检查器会拒绝那段代码。那一刻，脑海里某块拼图落定，有一种奇特的愉悦感。

编译器是在保护你

大多数初学者（包括我）的第一反应是：Rust 的编译器太烦了。它不让我做任何事。

但当你真正理解所有权模型之后，你会意识到：编译器每一次拒绝你，都是在阻止一个在运行时会出现的真实 bug——数据竞争、悬空指针、双重释放。

三条核心规则

所有权系统建立在三条简单规则之上：

1. 每个值都有一个所有者。
2. 同一时间只能有一个所有者。
3. 所有者离开作用域时，值被自动释放。

听起来简单，但组合起来产生了巨大的复杂性。让我用一段让我卡了很久的代码来说明：

fn main() {
    let s1 = String::from("hello");
    let s2 = s1;          // s1 的所有权 移动 给了 s2
    println!("{}", s1);   // ❌ 编译错误！s1 已失效
}

这在几乎所有其他语言里都是合法的。但 Rust 不允许，因为如果 s1 和 s2 都指向同一块内存，当它们都离开作用域时，内存会被释放两次——这就是「双重释放」漏洞。

借用：临时的访问权

当你不想转移所有权时，可以借用：

fn print_length(s: &String) {  // 借用，不取走所有权
    println!("长度: {}", s.len());
}

fn main() {
    let s = String::from("hello");
    print_length(&s);         // 传入引用
    println!("{}", s);        // ✅ s 依然有效
}

顿悟的那一刻

顿悟来自于一个思维转变：把所有权不看作「限制」，而看作「资源管理协议」。当你开始用这个视角读 Rust 代码，你会发现每一处 &、&mut、move 都在清晰地声明：谁负责这块数据的生命周期。

这种明确性，在大型系统中，是无价的。
